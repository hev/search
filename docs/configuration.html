Architecture

Configuration

hev search is configured entirely through environment variables. Either HEVSEARCH_STORAGE_URI (preferred) or the legacy HEVSEARCH_S3_BUCKET is required; everything else has a sensible default.

Core settings

VariableDefaultDescription
HEVSEARCH_STORAGE_URI required (or set HEVSEARCH_S3_BUCKET) Object-storage root that every namespace lives under. Accepts a URI of the form scheme://bucket[/prefix...] where scheme is s3 (any S3-compatible backend) or gs (native Google Cloud Storage). Namespace tables live at {root}/{namespace}/. Multi-segment prefixes are supported, e.g. s3://shared-bucket/tenants/acme/prod or gs://shared-bucket/tenants/acme/prod.
HEVSEARCH_S3_BUCKET legacy fallback Legacy S3-only form. Equivalent to HEVSEARCH_STORAGE_URI=s3://<bucket>. Both vars remain supported; if both are set they must agree, or startup fails.
HEVSEARCH_BIND 0.0.0.0:3000 Address and port for the HTTP listener. Use 127.0.0.1:3000 to restrict to localhost.
HEVSEARCH_MAX_BODY_BYTES 16777216 (16 MB) Maximum request body size for every JSON endpoint. Raise this when upserting large batches (multivector workloads in particular hit the ceiling fast, around 2 MB per row at 512 sub-vectors of dimension 128). Lower it to tighten the DoS surface in front of a trusted client.
RUST_LOG info Log level filter. Supports tracing directives like hevsearch_api=debug,hevsearch_core=trace.

Object storage backend

For S3-family backends, the HEVSEARCH_S3_* variables below configure the endpoint and credentials. For native Google Cloud Storage, set HEVSEARCH_STORAGE_URI=gs://<bucket> and one of the standard GOOGLE_* credential variables — no other hevsearch-specific configuration is required.

S3-family variables

For real AWS S3, you typically only need HEVSEARCH_STORAGE_URI. Credentials come from the standard AWS chain (instance profile, environment, ~/.aws/credentials), and the region follows AWS_REGION / AWS_DEFAULT_REGION if you do not set HEVSEARCH_S3_REGION explicitly.

VariableDefaultDescription
HEVSEARCH_S3_ENDPOINT unset (use AWS) Custom S3 endpoint URL. Set this for MinIO, R2, Tigris, DigitalOcean Spaces, or any S3-compatible backend. When set, path-style addressing and plain HTTP are automatically enabled. Not used for native GCS — set HEVSEARCH_STORAGE_URI=gs://... instead.
HEVSEARCH_S3_ACCESS_KEY unset S3 access key ID. Falls back to the standard AWS credential chain if not set.
HEVSEARCH_S3_SECRET_KEY unset S3 secret access key. Falls back to the standard AWS credential chain if not set.
HEVSEARCH_S3_REGION AWS_REGION / AWS_DEFAULT_REGION, else us-east-1 Region for the bucket. If unset, hev search falls back to the standard AWS_REGION, then AWS_DEFAULT_REGION, then us-east-1. Set it to pin the region explicitly. Some S3-compatible providers (R2, Tigris) accept auto. A region that does not match the bucket fails the request on backends that enforce region (real AWS S3); MinIO and most emulators ignore it.

Google Cloud Storage variables

Native GCS routing uses lancedb's gcs feature and the matching object_store::gcp client. Both authenticate from the standard Google environment variables; hevsearch forwards them unmodified.

VariableDefaultDescription
GOOGLE_APPLICATION_CREDENTIALS unset Path to an application-default-credentials file. May resolve to a service-account JSON, federated identity, or user creds.
GOOGLE_SERVICE_ACCOUNT_PATH unset Path to a service-account JSON file. Strictly service-account format only — distinct from GOOGLE_APPLICATION_CREDENTIALS.
GOOGLE_SERVICE_ACCOUNT_KEY unset Inline service-account JSON, as a single string. Useful for environments where mounting a file is awkward (some serverless platforms). Redacted from Debug output.

AWS S3

For production AWS deployments, let the standard credential chain handle authentication (instance profiles, ECS task roles, or ~/.aws/credentials).

HEVSEARCH_STORAGE_URI=s3://my-hevsearch-bucket
HEVSEARCH_S3_REGION=eu-west-1

HEVSEARCH_S3_ACCESS_KEY=<r2-access-key>
HEVSEARCH_S3_SECRET_KEY=<r2-secret-key>

HEVSEARCH_S3_ACCESS_KEY=<tigris-access-key>
HEVSEARCH_S3_SECRET_KEY=<tigris-secret-key>
HEVSEARCH_S3_REGION=auto

DigitalOcean Spaces

HEVSEARCH_STORAGE_URI=s3://hevsearch-spaces
HEVSEARCH_S3_ENDPOINT=https://<region>.digitaloceanspaces.com
HEVSEARCH_S3_ACCESS_KEY=<spaces-access-key>
HEVSEARCH_S3_SECRET_KEY=<spaces-secret-key>

Google Cloud Storage (native)

Use the gs:// scheme. The GCS S3-interop endpoint is a separate path that silently drops If-None-Match: * and is not supported — do not reach for it.

HEVSEARCH_STORAGE_URI=gs://my-hevsearch-bucket
GOOGLE_APPLICATION_CREDENTIALS=/etc/hevsearch/gcp-sa.json
Provider validation
hev search is validated against AWS S3, MinIO, Cloudflare R2, Tigris, DigitalOcean Spaces, and native Google Cloud Storage with 100-run concurrent-writer stress tests. Backblaze B2 and the GCS S3-interop endpoint are not supported. See the compatibility matrix in the project README for the per-provider details.

Cache settings

The foyer hybrid cache has two tiers: a RAM tier for the hottest data and an NVMe tier for overflow. Both tiers are shared across all namespaces.

VariableDefaultDescription
HEVSEARCH_CACHE_MEMORY_BYTES 67108864 (64 MB) RAM tier capacity in bytes. This is the L1 cache; hits here are sub-microsecond.
HEVSEARCH_CACHE_NVME_PATH /tmp/hevsearch-cache Directory for the foyer NVMe-tier block file. In production, point this at a fast local SSD. The Docker image defaults to /var/lib/hevsearch/cache.
HEVSEARCH_CACHE_NVME_BYTES 268435456 (256 MB) NVMe tier capacity in bytes. This is the L2 cache; hits here avoid the object-storage round-trip but are slower than RAM.

Cache sizing guidance

The right cache size depends on your working set. A cached result for a 10-result query with 1536-dim vectors is roughly 62 KB. Some rules of thumb:

Working setRAM tierNVMe tier
Development / testing64 MB256 MB
Small production (hundreds of cached queries)256 MB1 GB
Large production (thousands of cached queries)1 GB10 GB
Cache is ephemeral
The cache is a performance optimisation, not durable storage. Data in the configured object store is always the source of truth. If the cache is lost (restart, crash, disk failure), hev search repopulates it transparently on the next query.

Object cache

The object cache is a separate, optional layer from the result cache above. It is a local byte-range cache that sits under the storage engine: on a cache miss the engine still reads the namespace's data and index files from object storage, but those reads are served from local disk on subsequent queries. It accelerates cold and novel queries (different filters, new vectors) that the result cache cannot help, because they never hit object storage twice for the same bytes.

It only caches immutable, uniquely-named objects (data fragments and index files, which are written once and never mutated). Manifests, version pointers, and transaction logs are always read straight through, so a write, delete, compaction, or index build is reflected immediately — the cache cannot serve stale data. It is disabled by default.

VariableDefaultDescription
HEVSEARCH_OBJECT_CACHE_ENABLED false Enable the byte-range object cache. When off, every object-storage read goes straight to the backend.
HEVSEARCH_OBJECT_CACHE_DIR /tmp/hevsearch-object-cache Directory for cached object bytes. In production, point this at a fast local SSD/NVMe. Existing files are scanned and counted toward the capacity on startup, so the cache survives a restart within its byte budget.
HEVSEARCH_OBJECT_CACHE_BYTES 10737418240 (10 GB) Maximum bytes of cached object data held on disk. When full, least-recently-used entries are evicted. The cap is enforced across restarts (small per-entry size sidecars are negligible additional overhead).
HEVSEARCH_OBJECT_CACHE_MAX_ENTRY_BYTES 268435456 (256 MB) Largest single read the cache will buffer and store. A read above this is streamed straight through and not cached, so one large fragment read cannot spike memory by its full size. Lower it on memory-constrained hosts; raise it to cache larger fragments whole.
Result cache vs object cache
The result cache stores whole query results and turns a repeated query into zero object-storage reads. The object cache stores raw file byte ranges and makes the first execution of a new query cheaper the next time similar data is read. They are complementary and can be enabled together.

Trusted internal service

There are no auth, rate-limit, metrics-token, or proxy-trust environment variables. RFC 0002 removed that edge surface from the engine: hev search runs open inside a trusted network, and Layer owns authentication, tenant authorization, rate limiting, and proxy-header trust.

For production, expose hev search only to Layer or operator tooling. In Kubernetes, enforce that posture with a NetworkPolicy; in other environments, use the equivalent private subnet or firewall rule. Do not expose the engine listener directly to untrusted clients.

Docker Compose override

To customise settings for local development, create a docker-compose.override.yml:

# docker-compose.override.yml
services:
  hevsearch:
    environment:
      HEVSEARCH_CACHE_MEMORY_BYTES: "268435456"   # 256 MB RAM cache
      HEVSEARCH_CACHE_NVME_BYTES: "1073741824"    # 1 GB NVMe cache
      RUST_LOG: "hevsearch_core=debug,hevsearch_api=debug"

Full example

A complete production configuration against AWS S3:

# Production environment
HEVSEARCH_STORAGE_URI=s3://prod-hevsearch-vectors
HEVSEARCH_S3_REGION=eu-west-1
HEVSEARCH_BIND=0.0.0.0:3000
HEVSEARCH_MAX_BODY_BYTES=33554432            # 32 MB body limit for larger batches
HEVSEARCH_CACHE_MEMORY_BYTES=268435456
HEVSEARCH_CACHE_NVME_PATH=/var/lib/hevsearch/cache
HEVSEARCH_CACHE_NVME_BYTES=10737418240
RUST_LOG=info
