Deployment

Run hev search locally with Docker Compose or deploy to production as a standalone container.

Local development

The Docker Compose file launches MinIO (local S3) alongside the hev search API. No host Rust toolchain is required.

git clone https://github.com/gordonmurray/hevsearch
cd hevsearch
docker compose up --build

This starts three services:

ServicePortPurpose
minio9000 (S3 API), 9001 (console)S3-compatible object storage
minio-init-One-shot: creates the hevsearch bucket
hevsearch3000hev search API server

MinIO console: http://localhost:9001 (credentials: minioadmin / minioadmin)

Running just MinIO

If you want to run hev search outside Docker (e.g. during development with a host Rust toolchain), start only MinIO:

# Start MinIO only
docker compose up -d minio minio-init

# Run hev search directly
HEVSEARCH_STORAGE_URI=s3://hevsearch \
HEVSEARCH_S3_ENDPOINT=http://127.0.0.1:9000 \
HEVSEARCH_S3_ACCESS_KEY=minioadmin \
HEVSEARCH_S3_SECRET_KEY=minioadmin \
  cargo run -p hevsearch-api

Production Docker

The included Dockerfile is a multi-stage build that produces a minimal production image.

Build the image

docker build -t hevsearch-api .

Image details

StageBase imagePurpose
Builderrust:1.94-bookwormCompiles the release binary with protobuf support
Runtimedebian:bookworm-slimMinimal image with just ca-certificates for TLS

Run the container

docker run -d \
  --name hevsearch \
  -p 3000:3000 \
  -e HEVSEARCH_STORAGE_URI=s3://my-production-bucket \
  -e HEVSEARCH_S3_REGION=eu-west-1 \
  -e HEVSEARCH_CACHE_MEMORY_BYTES=268435456 \
  -e HEVSEARCH_CACHE_NVME_BYTES=10737418240 \
  -v hevsearch-cache:/var/lib/hevsearch/cache \
  hevsearch-api

For native GCS, swap the URI scheme and mount a service-account JSON: -e HEVSEARCH_STORAGE_URI=gs://my-production-bucket -e GOOGLE_APPLICATION_CREDENTIALS=/etc/gcp-sa.json -v /etc/gcp-sa.json:/etc/gcp-sa.json:ro. Everything else stays the same.

NVMe cache volume
Mount a fast local SSD at /var/lib/hevsearch/cache for the NVMe tier. The cache is ephemeral: losing it only means cache misses until it warms up again. For best performance, use a tmpfs or NVMe-backed volume.

AWS deployment

For production on AWS, hev search works best with IAM-based credentials (instance profiles, ECS task roles, or IRSA for EKS). No access keys needed.

Prerequisites

  1. An S3 bucket with public access blocked
  2. An IAM role with s3:GetObject, s3:PutObject, s3:DeleteObject, s3:ListBucket on the bucket
  3. A compute environment (ECS, EKS, EC2) with the IAM role attached

Minimal IAM policy

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-hevsearch-bucket",
        "arn:aws:s3:::my-hevsearch-bucket/*"
      ]
    }
  ]
}

ECS task definition (excerpt)

{
  "containerDefinitions": [
    {
      "name": "hevsearch",
      "image": "hevsearch-api:latest",
      "portMappings": [{"containerPort": 3000}],
      "environment": [
        {"name": "HEVSEARCH_STORAGE_URI", "value": "s3://my-hevsearch-bucket"},
        {"name": "HEVSEARCH_S3_REGION", "value": "eu-west-1"},
        {"name": "HEVSEARCH_CACHE_MEMORY_BYTES", "value": "268435456"},

Trusted internal service

hev search is an open internal engine. It does not ship bearer-token auth, admin scopes, metrics tokens, rate limiting, or proxy-header trust. Layer owns the edge: authentication, per-tenant authorization, request shaping, and any rate limits happen before traffic reaches the engine.

Production deployments should make direct engine reachability impossible from untrusted clients. In Kubernetes, allow ingress to the hev search pod only from Layer and approved operator namespaces with a NetworkPolicy. For Docker or VM deployments, use private networking or firewall rules with the same intent.

# engine call from the trusted network
curl -X POST http://hevsearch:3000/ns/production/upsert \
  -H 'Content-Type: application/json' \
  -d '{"rows": [...]}'

Health checks

Use GET /health for both liveness and readiness probes. The endpoint returns 200 with body ok as soon as the server is listening.

Docker Compose

healthcheck:
  test: ["CMD-SHELL", "curl -f http://localhost:3000/health || exit 1"]
  interval: 10s
  timeout: 3s
  retries: 3

Kubernetes

livenessProbe:
  httpGet:
    path: /health
    port: 3000
  initialDelaySeconds: 5
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /health
    port: 3000
  initialDelaySeconds: 5
  periodSeconds: 5

Cache warmup after deployment

After a fresh deployment or restart, the cache is empty. Use the /ns/{ns}/warmup endpoint to pre-populate it with your most common queries:

curl -X POST http://localhost:3000/ns/production/warmup \
  -H 'Content-Type: application/json' \
  -d '{
    "queries": [
      {"vector": [1.0, 0.0, ...], "k": 10},
      {"vector": [0.0, 1.0, ...], "k": 10}
    ]
  }'

This returns immediately (202) and runs the queries in the background, populating the exact cache as it goes. If your production traffic uses semantic caching, include the same semantic_cache block on eligible single-vector warmup queries so the in-memory semantic sidecar is seeded too. Monitor hevsearch_cache_misses_total to track warmup progress and hevsearch_semantic_cache_hits_total / hevsearch_semantic_cache_rejections_total to confirm the semantic layer is behaving as expected.

Recommended operational setup

  • Network boundary: keep hev search reachable only from Layer and approved operator tooling
  • Prometheus scraping: scrape /metrics at 15s intervals
  • Log aggregation: hev search logs structured JSON via tracing. Ship to your log aggregator.
  • Cache volume: use a fast local SSD or NVMe, not networked storage
  • Compaction schedule: run /ns/{ns}/compact periodically for write-heavy namespaces
