# Engine RFCs

Design proposals for **hev search** (the engine behind hev layer; a hard fork of
firnflow developed as ours). These are **engine-scoped** — search, storage,
indexing, caching, metadata, filter, facets, tokenization. The *edge* view (how
Layer operates this engine) lives in `../../../layer/docs/rfcs/` — RFC 0086 there
is the Layer-side twin of this engine.

This series is numbered **independently** of Layer's series; an engine `0001` and
a Layer `0001` are unrelated. Apply the engine/edge test (`AGENTS.md`) before
adding one: if it's edge (auth, tenancy, rate limiting), it belongs in `../layer`,
not here.

| RFC | Title | State |
|-----|-------|-------|
| [0001](0001-alyze-tokenizer.md) | alyze tokenizer for full-text search | Implemented (#5). LanceDB's built-in FTS analyzer replaced with [alyze](https://github.com/turbopuffer/alyze) `=0.1.5` (`word_v4`) run as an external analyzer: one engine-owned analyzer (`analyzer.rs`) derives the reserved `text_tok` column on every write and analyzes every query; the FTS index is a passthrough whitespace tokenizer + BM25 over `text_tok`, the analyzer id is captured in schema metadata, and `POST /ns/{ns}/fts-index` backfills pre-existing namespaces. Motivated by **Turbopuffer wire parity** — Turbopuffer's BM25 is `word_v4`, the dialect Layer translates onto this engine. Engine-local (hard fork; not contributed upstream). |
| [0002](0002-remove-auth.md) | Remove auth, rate limiting, and proxy-header trust | Draft. The flagship subtractive removal: delete the `auth`/`rate_limit` modules, the `HEVSEARCH_API_KEY`/`ADMIN`/`METRICS_TOKEN`/`RATE_LIMIT_*`/`TRUST_PROXY_HEADERS` config, and the `subtle`/`governor`/`tower_governor` deps (~900–1000 lines). Layer is the auth boundary and the engine's only client; the engine is a trusted internal service behind a `NetworkPolicy`. Keeps physical namespace-prefix isolation (engine, not authz). Engine-local + permanent. |
| [0003](0003-row-delete.md) | Per-row delete (by id, then by filter) | Implemented. **Additive.** `POST /ns/{ns}/delete` accepts exactly one of `ids` or `filter`, compiles to LanceDB `Table::delete`, invalidates caches via the normal table-version path, and is wired through Layer's search backend delete mapping. |
| [0004](0004-fuzzy-fts.md) | Fuzzy / typo-tolerant full-text search | Implemented (#4). `fuzzy.max_edit_distance` (`0`/`1`/`2`/`"auto"`) on `/query`: per-token expansion with a length-keyed ladder (1–5 chars exact, 6–8 d=1, 9+ d=2; a fixed value caps the ladder), so short tokens stay exact instead of match-flooding. Fusion stays in the engine. The analyzer-parity gap it named is closed by RFC 0001 (#5): fuzzy now expands alyze tokens against the unstemmed `text_tok` surface. |
| [0005](0005-arbitrary-row-ids.md) | Arbitrary (string) row ids | Draft. **Additive.** Rows are keyed by `u64` end to end (schema, merge key, BTree, list cursor); Layer's document model is string-keyed. Add a `Utf8` id type fixed per namespace at first write (like vector kind/dim), so string ids land without a lossy gateway-side surrogate map (RFC 0086). |
| [0006](0006-configurable-distance-metric.md) | Configurable distance metric | Draft. **Additive.** The metric is hardcoded by vector kind (L2 single / cosine multivector) with "no override"; add a per-namespace `distance_metric ∈ {l2, cosine, dot}` fixed at creation, defaulting to today's behavior, so a cosine/dot Turbopuffer namespace is faithfully mirrorable without edge-side normalization and the operator's `distanceMetric` validation (RFC 0086) becomes real. |
| [0007](0007-point-fetch-and-nearest-to-id.md) | Point-fetch by id and nearest-to-stored-id query | Draft. **Additive, read-path only.** The engine can ANN on a caller-supplied vector but not on the vector of a row it already stores — there is no `nearest_to_id`, so Layer's `recommend` / "more like this" (`shop`) round-trips the seed vector out and back. Add `nearest_to_id` to `/query` (resolve the seed vector internally via the existing `id` BTree + `only_if`, ANN with it, exclude the seed); a point-read `GET /ns/{ns}/doc/{id}` is named but optional and boundary-fenced (point-fetch-by-id stays Layer's document cache). The headline is the vector search, which the cache cannot serve. |
| [0008](0008-bulk-ingest-upsert-and-pipeline.md) | Bulk ingest — upsert, import, and the Layer Pipeline write path | Draft, exploration (options kept open). Frames the write-path design space: idempotent `/upsert` (JSON, latest-write-wins) vs bulk `/import` (Arrow IPC, insert-only, async). The gateway's `VectorStoreClient.write_rows` maps to `/upsert`; `/import` has no natural inbound mapping, so a bulk first load through Layer would otherwise pay the JSON path. Catalogs the options (Pipeline drives `/import` directly, a trait `bulk_write`, unify import under upsert semantics, or accept JSON) and their constraints. Engine half; orchestration is Layer's Pipeline (RFC 0086). Gated on RFC 0005 (string ids) for the real workload. |
| [0009](0009-pluggable-vector-index-type.md) | Pluggable vector index type (HNSW) | Draft. **Additive.** The vector index is hardcoded to IVF_PQ (`create_index` always builds `Index::IvfPq`; the handler rejects any `kind != "ivf_pq"`), and the query path exposes only `nprobes` — no candidate re-ranking. Turn the inert `IndexRequest.kind` field into a real selector over the **IVF_HNSW family** (`ivf_hnsw_sq`/`ivf_hnsw_pq`) and plumb the `ef` / `refine_factor` query knobs — all already in the pinned `lancedb 0.29.0`, no new ANN code, no version bump. `refine_factor` targets the PQ recall ceiling (recall@100 flat at 0.707 across nprobes in the benches). Single-vector only in v1; SPFresh framed as a separate "outgrow LanceDB" follow-up. Engine-local. |
| [0010](0010-late-interaction-storage.md) | Late-interaction storage — half-precision token bags | Draft. **Additive.** ColBERT-style late interaction already works (a `Multivector` namespace stores a per-token bag and scores it via LanceDB's MaxSim plan, `manager.rs:1664-1693`), but the token sub-vector is **hardcoded to `Float32`** (`manager.rs:675`), so every token costs `dim×4` bytes — the ~500 KB/entry footprint RFC 0086 names as the multivector cost. Storage *is* the late-interaction cost (one vector per token), so add a per-namespace **token precision** (`f32` default, `f16`) fixed at first write like dim/kind/metric, halving the footprint at the precision the model already emits, riding LanceDB's native `f16` vectors (no version bump). Adopts **Iso-ModernColBERT** (128-dim, cosine, f16, Apache-2.0 — matches the forced-cosine multivector constraint) as the reference model the BEIR multivector harness benchmarks against (incumbent baseline: `lightonai/LateOn` f32). int8/4-bit PLAID named as the next rung; pooled-FDE two-stage framed as a separate decision (the engine does MaxSim natively, unlike Turbopuffer in RFC 0049). Edge twin: Layer RFC 0089. Engine-local. |
| [0011](0011-recall-and-build-benchmark-harness.md) | Recall + build benchmark harness | Draft. **Additive tooling + the build-memory knobs it surfaces.** There is no in-repo recall harness — `hevsearch-bench` is latency/cache only on synthetic vectors, and the `recall@k`/`ndcg` numbers in `bench/results/` came from an external, unversioned tool. Version a recall harness in `hevsearch-bench` driven against the engine REST over **public ground-truth ANN datasets** (SIFT/GIST/Deep, exact-NN ground truth — isolates index recall from embedding quality, unlike BEIR's 0.83 ceiling), structured around the four index requirements: **filtered recall at varying selectivity**, **freshness** (insert→query-before-optimize→fold), **recall-at-latency**, and **build cost under a fixed cgroup RAM limit**. The constrained-build axis records peak RSS / OOM point / spill-bytes / wall-clock per (variant × size × budget) and specifies the build-memory knobs it justifies (NVMe build-scratch — today NVMe is cache-only, risking the tmpfs trap; a build-RAM budget + bounded partition concurrency; pq/sq variant as a memory dial; build off the serving process, which today is an in-process `tokio::spawn` that can OOM the serving pod). The index-variant sweep is gated on RFC 0009; the store-vs-store / Turbopuffer comparison is the anonymized edge twin in `../layer` (§2.4), reusing this harness's loaders + ground-truth. Engine-local. |
| [0012](0012-lsn-read-consistency.md) | LSN-based read consistency | Draft, proposal. **Additive read-path + write-response change.** Layer's consistency watermark is a Turbopuffer-shaped timestamp (`_hevlayer_upserted_at`) that the engine **strips on write** (`clients/search.rs:355–361`), so the gateway's injected stable-read predicate is a silent no-op that fails open — folded here as the motivating defect. The engine already tracks a durable monotonic per-namespace commit version (`table_version`, `result.rs:208`; cache-aside already keys on it) — that **is** an LSN. Surface it: writes return the LSN, reads accept `lsn` + `consistency` (`indexed`/`balanced`/`strong`, `strong` waits until `applied >= lsn`), mirroring [TopK](https://docs.topk.io/concepts#lsn-based-consistency). Unlocks exact read-your-writes and streaming ingest (every commit gets a sequence number, not just batches). Flips the matrix row from `heuristic`/"no LSN" to exact. Edge twin: Layer RFC 0086. Answers RFC 0008's bulk-consistency question; touches RFC 0011's freshness axis. Engine-local. |
| [0013](0013-small-write-delta-persistence.md) | Delta-friendly small writes — group commit, WAL buffering, auto-compaction | Draft, exploration (options kept open). Every `/upsert` batch is its own Lance commit (one `merge_insert` per call, `manager.rs:1054-1061`) — a tiny fragment + manifest + txn regardless of batch size, with documented decay (~2–6 docs/s per-batch upsert vs ~2,830 docs/s single-commit import, `bench/results/beir_multivector_objcache.md:372-390`) and **manual-only** `/compact` (no auto-policy; old-version cleanup skipped on import). Catalogs the options: **(A)** group commit coalescing concurrent upserts (LSN-neutral, cheap first step), **(B)** engine-side WAL/memtable folding into checkpoint commits (the streaming end-state; the LodeDB delta-persistence shape — O(changed) commits, WAL + checkpoint, atomic generation pointer — but forces the where-does-the-WAL-live and LSN-at-append-or-checkpoint questions), **(C)** auto-compaction policy (table stakes under every option), **(D)** push batching to Layer (boundary error, not favored). Likely sequencing C → A → B. Interacts with RFC 0012 (streaming on-ramp; LSN assignment), 0011 (sustained-trickle scenario), 0008. Engine-local. |
| [0014](0014-exact-knn-query-mode.md) | Exact KNN as an explicit query mode | Draft. **Additive, read-path only.** Brute-force scan exists only *by accident* — namespaces with no vector index flat-scan (`manager.rs:1520-1524`); once IVF_PQ is built there is no way to request the exact answer (`nprobes` is the only knob). Add `exact: true` on `/query`, riding `VectorQuery::bypass_vector_index()` already in the pinned `lancedb 0.29.0` (`query.rs:1184`, no version bump) — prefilter/hybrid/MaxSim semantics unchanged, `exact` joins the result-cache key. Headline consumer is RFC 0011: in-engine ground truth (same metric, same filter path) makes recall self-calibrating on arbitrary corpora and unlocks filtered-recall ground truth; also the escape hatch for the 0.707 PQ recall ceiling until RFC 0009's `refine_factor` lands, and free 100% recall on small namespaces (exact-on-request, not LodeDB's exact-by-default). Engine-local. |
